# 首页布局与交互设计规范 (Homepage Layout & Interaction Spec)

## 1. 核心目标

将首页打造为类似 iOS 主屏幕的体验，具有固定的全屏网格布局，消除所有垂直滚动（Vertical Scroll），并采用整屏滑动的水平翻页（Horizontal Pagination）来展示更多的组件（Widget）。

## 2. 核心规则与约束

- **无垂直滚动（No Vertical Scroll）**：无论在何种屏幕尺寸或极端比例下，首页内容必须完全容纳在计算好的视口高度（屏幕高度减去顶部导航栏等占据的高度）内。
- **整屏翻页（Carousel/Snap Pagination）**：第二屏或第三屏的切换应当是整页滑动的，像 Carousel 或 iOS 主页面那样，而不是连续的自由水平滚动。
- **绝对居中对齐（Absolute Centering）**：在遇到屏幕过宽或过高的极端情况时，整个计算好的网格应当在可用视口内水平和垂直绝对居中（目前基于屏幕 `stageW` 和 `stageH` 动态计算 `pageOffsetX` 和 `pageOffsetY` 实现）。8列布局在大屏下会根据此逻辑在屏幕正中央显示。

## 3. 断点与网格定义 (Breakpoints & Grid System)

### 3.1 大屏设备 (iPad横屏/竖屏, MacBook及更大屏幕)

- **列数 (Columns)**：固定为 `8列`。
- **行数 (Rows)**：固定为最多 `4行`。
- **单屏容量**：`8 x 4 = 32个基础格子`。
- **响应式缩放策略 (等比缩放)**：
  - **屏幕过宽、高度不足 (如超宽带鱼屏)**：维持 8 x 4 的网格结构，基于屏幕**高度**来计算格子大小，确保4行不会溢出屏幕，网格在水平方向居中。
  - **屏幕过高、宽度不足 (如竖屏90度显示器或iPad竖屏)**：维持 8 x 4 的网格结构，基于屏幕**宽度**来计算格子大小，确保8列不会溢出边框，网格在垂直方向居中。

### 3.2 手机/小屏设备 (Mobile Screens)

- **列数 (Columns)**：固定为 `4列`。
- **行数 (Rows)**：根据实际可用高度和最佳实践待定（通常为 6 到 8 行），但同样要遵守“网格高度不能超出屏幕”的原则。
- **单屏容量**：`4 x MaxRows`。
- **响应式缩放策略**：与大屏类似，基于较短的一边来约束格子的大小，永远不会出现垂直滚动条。

## 4. 当前 `src/app/page.tsx` 实现情况分析

### 4.1 垂直方向溢出与滚动问题

- **当前状态**：首页使用的是 `overflow-y-auto no-scrollbar`，网格格子（cellW）仅仅基于 `stageW / cols` 来计算，并没有对高度和行数做严格限制。
- **存在的问题**：当屏幕高度较小（宽高比大），且有多个小组件被摆放到靠下位置时，依然会导致页面出现垂直滚动。
- **优化方案**：
  1. 引入最大行数常量（大屏为4，手机端考虑6-8）。
  2. 格子大小逻辑调整：

     ```typescript
     // 现有纯宽度计算
     let cw = availableWidth / cols;
     
     // 必须引入高度计算：
     let ch = availableHeight / rows;
     
     // 取最小的一个来做等比缩放，确保无论如何都在容器内
     let finalCellSize = Math.min(cw, ch);
     ```

### 4.2 翻页与水平拓展

- **当前状态**：通过原生的 CSS `snap-x snap-mandatory` 实现水平滚动。虽然带有一定 Snap 效果，但是在体验上容易感觉像是水平长画卷滚动。
- **存在的问题**：目前只限制了 X 坐标不超过 `gridCols * pageCount`，且依靠 DOM 元素的实际宽度将内容撑开。有时候跨屏拖拽会导致排版诡异。
- **优化方案**：
  保持每一屏 `100vw` 宽度绝对分隔，或者继续使用 Snap，但要确保 Snap 点精确为 `100vw`。同时，在重新排版（`resolveOverlap`）时，强制控制 Y 轴的值不得大于 `MAX_ROWS - 1`，如果超过了，就强制溢出到下一屏（X 轴增加到下一屏的列）。

### 4.3 居中显示

- **当前状态**：当前的 `toLeft` 函数已经做了部分针对宽度的居中 `pageOffsetX = (stageW - gridWidth) / 2`，但缺少针对高度的居中逻辑。
- **优化方案**：需要计算 `pageOffsetY = (stageH - gridHeight) / 2`，在渲染每个格子以及渲染高亮、ghost格子时应用。

## 5. 建议执行步骤 (Next Steps)

1. **重构 Dimensions Calculation**：在 `useMemo` 中修改 `cellW, cellH` 的计算逻辑，保证依赖视口高度和行数限制。
2. **强制分页逻辑与边界限制**：修改组件重排（`resolveOverlap`）和拖拽判断的约束，保证组件不能再放置到4行以下，而是转入另一屏。
3. **增加全屏容器的严格样式**：使用 `overflow: hidden` 去掉大容器的各类自动滚动。
4. **计算垂向居中偏移 (Vertical Centering)**：基于新算出的 cellSize，计算整体高度，得出顶底部的 margin/padding 并在 `toTop()` 转换中提供支持。

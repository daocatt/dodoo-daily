# DoDoo Daily - 数据库设计评审 (Code Review)

**评审对象**：`src/lib/schema.ts`  
**评审时间**：2026-03-09  
**评审重点**：数据一致性、可扩展性、安全合规、Sqlite 适配性

---

## 1. 核心架构亮点 (Key Strengths)

* **业务模块化**：数据表清晰地划分为用户系统、任务系统、情绪管理、日记展览及商店系统，模块边界清晰。
* **Snapshot (快照) 设计**：`Purchase`（购买）表中冗余了商品名称、图标和描述。这确保了日后即使原始 `ShopItem` 被修改或软删除，用户的历史订单信息依然完整可追溯。
* **聚合统计分离**：将账户数值（星星、金币）从 `Users` 表剥离至 `AccountStats` 表。有效避免了高频数值变动锁定主用户信息表，提升了系统吞吐量。
* **多层级日志系统**：针对金币和多种星星变动设计了统一的流水表（`AccountStatsLog`）及分类流水表，具备极高的可审计性，方便对账。
* **生产级软删除**：核心表通过 `isDeleted` 和 `isArchived` 支持逻辑删除，确保数据生命周期的安全管理。

---

## 2. 识别到的问题与优化建议 (Improvements)

### A. 任务系统的冗余 (Task vs AssignedTask)

* **现状**：`Task` (个人任务) 与 `AssignedTask` (分配任务) 字段高度平行。
* **挑战**：未来若需增加附件、权重或多阶段状态，需要在两张表维护相同逻辑。
* **优化**：合并为统一的 `Tasks` 表，通过 `type` 字段区分模式。将 `assignerId` 和 `assigneeId` 设为通用外键（个人任务时两者相同），以减少代码冗余并便于跨类型统计。

### B. 周期性任务的确切定义

* **现状**：`isRepeating` 仅为布尔开关。
* **挑战**：目前的结构无法支持复杂的周期逻辑（如：每周二执行、每月 1 号执行等），也难以自动生成下一实例。
* **优化**：建议增加 `cronExpression` (字符串) 或 `repeatInterval` (JSON) 字段，存储具体的执行规则。

### C. 日记系统的多媒体扩展性

* **现状**：`Journal` 表使用 `imageUrl` 和 `imageUrls` (JSON 字符串) 混合模式。
* **挑战**：对于高性能查询或未来支持音视频混合排版，JSON 解析会成为瓶颈。
* **优化**：建立 `JournalMedia` 一对多表，关联具体的媒体资源。既方便统计媒体大小，也利于实现更加复杂的相册功能。

### D. 时间戳一致性 (Sqlite 适配)

* **现状**：`createdAt` 使用 Sqlite 原生秒级时间戳，而 `updatedAt` 配合 Drizzle API 生成 JS Date (毫秒)。
* **挑战**：前端渲染时可能出现混淆（某些除以 1000，某些不用）。
* **优化**：统一使用 Drizzle 的 `mode: "timestamp_ms"`，或统一在 SQL 层面处理毫秒精度，确保全链路时间单位对齐。

### E. 数据冗余风险

* **现状**：`Journal` 表冗余了 `authorRole`。
* **优化**：除非是为了记录“发布时的瞬时身份”，否则建议通过 Join 用户表获取动态身份。

---

## 3. 下一阶段扩展建议 (Roadmap)

1. **引入 Drizzle Relations API**：在 Schema 文件中显式定义表关系，可极大简化前端的联表查询代码，提升开发效率。
2. **索引优化**：目前仅主键自带索引。建议在 `userId`, `authorId`, `creatorId` 等常用查询过滤字段上显式增加索引（`index("idx_name").on(...)`）。
3. **约束加固**：在流转较为严格的数值变动（如金币扣减）上增加数据库层面的 `check` 约束，确正数值不为负。

---
**评审结论**：当前设计处于 **【健康】** 状态，完全能够支撑 MVP 到中型规模的业务。通过上述局部重构，系统将具备更强的工业级鲁棒性。
